# Contributing to Bindr

Thanks for your interest in contributing! Bindr turns Media Library PDFs into
interactive flipbooks — self-hosted, privacy-friendly, no external services.

## Development setup

Requirements: Node.js ≥ 18, Composer, PHP ≥ 7.4, and a local WordPress
(≥ 6.0) install for manual testing.

```bash
npm install          # JS toolchain (@wordpress/scripts)
composer install     # PHPCS (WordPress Coding Standards)
npm run build        # compile block + viewer, copy vendor libs, build i18n
```

`npm run start` watches and rebuilds the JS/SCSS bundles during development.

To test in WordPress, zip the runtime files and upload via
Plugins → Add New → Upload Plugin:

```bash
zip -r wp-bindr.zip wp-bindr.php uninstall.php readme.txt LICENSE LIBRARIES.md \
  includes templates languages build -x "*.DS_Store"
```

## Project layout

| Path         | What lives there                                              |
| ------------ | ------------------------------------------------------------- |
| `wp-bindr.php` | Plugin bootstrap: constants, autoloader, activation hooks    |
| `includes/`  | PHP classes (CPT, block/shortcode, viewer, analytics, admin)  |
| `src/`       | JS/SCSS sources — compiled into `build/` (never edit `build/` by hand) |
| `templates/` | The blank-canvas full-page reading template                   |
| `languages/` | Translation catalogs (regenerated by `npm run build`)         |
| `tools/`     | Build helpers: vendor copy, POT extraction, MO/JED compile    |

Compiled assets in `build/` are committed so the repo is installable as-is.
Rebuild and commit them together with the source change that produced them.

## Coding standards

- **PHP** follows the WordPress Coding Standards — run `composer lint`
  (`composer lint:fix` auto-fixes what it can).
- **JS** uses the WordPress ESLint preset — run `npm run lint:js`.
- All user-facing strings are translatable in the `wp-bindr` text domain.
  Terminology: **Bindr** is the product name; a **flipbook** is what it
  produces. The shortcode is `[bindr]`.
- Everything the viewer outputs is escaped; database access goes through
  `$wpdb->prepare()`.

## Naming conventions (stable identifiers)

These are public API — renaming them breaks existing sites, so they must not
change casually:

- Post type: `bindr_flipbook` (URL slug `/bindr/…`)
- Shortcode: `[bindr id="123"]`
- Block: `bindr/flipbook`
- Hooks: `bindr_*` (e.g. `bindr_pdf_url`, `bindr_viewer_config`, `bindr_flipbook_saved`)
- Tables: `{$prefix}bindr_events`, `{$prefix}bindr_daily`
- Settings option: `bindr_settings`; post meta prefixed `_bindr_`

## Database changes

Schema lives in `Bindr_Analytics::create_tables()` and runs through
`dbDelta()`. Any schema change must bump `BINDR_DB_VERSION` in
`wp-bindr.php` and, if `dbDelta()` cannot express the change (renames,
primary-key changes), add an explicit migration step to
`Bindr_Analytics::maybe_upgrade_db()`.

## Privacy rules for analytics

No cookies, no PII, nothing identifying stored or sent anywhere. Readers are
counted with a salted daily hash computed server-side. Keep it that way — a
change that stores IPs, user agents, or cross-day identifiers will not be
merged.

## Pull requests

1. Fork, branch from `main`, keep the change focused.
2. Run `composer lint` and `npm run lint:js`; rebuild `build/` if you touched
   `src/`.
3. Test the three render paths when relevant: block, shortcode, and the
   full-page reading mode — plus mobile width (single-page engine) and
   keyboard navigation.
4. Describe *why* in the PR body, not just what.

## Reporting issues

Open a GitHub issue with WordPress/PHP versions, the theme, reproduction
steps, and — for viewer bugs — the browser and a sample PDF when possible.
